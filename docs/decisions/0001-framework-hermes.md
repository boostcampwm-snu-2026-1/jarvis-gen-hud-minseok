# ADR 0001 — 에이전트 프레임워크: Hermes

> 상태: **확정** · 날짜: 2026-06-04 · 관련: `docs/decisions/0002-hermes-feasibility.md`

## 맥락

JARVIS 프로젝트의 두뇌(브레인)를 무엇으로 할지. 평가 대상은 **generative HUD(React 프론트)**이고 브레인·음성·툴은 위임한다("0부터 짓는 건 HUD 하나", 백엔드 늪 회피). 제약: 마감 6/14, 솔로, 원격 개발, **프로바이더 OpenAI + Anthropic만**, 배포는 워크스테이션 서버 + 노트북 브라우저(키 서버측).

## 요구사항 (수렴본)

브레인이 충족해야 할 것:

1. 코히어런트 JARVIS 페르소나 (서버측 단일출처)
2. 영속 메모리
3. 스킬·툴 사용 (가능하면 합성)
4. **역할별 모델 하이브리드** — 빠른 메인(음성·dispatch) + 강한 모델(추론·코딩), **단일 브레인**
5. 낮은 TTFT (음성 즉답)
6. **백그라운드/비동기 실행** ("돌려놓고 계속 응답") + 진행·완료 알림
7. 오케스트레이션/dispatch (서브태스크·병렬)
8. **OpenAI 호환 `/v1`** (프론트 연결)
9. HUD용 콘텐츠/JSX 스트리밍
10. 음성+HUD 동시성 (오케스트레이터가 멀티플렉싱)
11. **deterministic 계산** (환각 수치 금지)
12. 보안 (키 서버측)

## 결정

**Hermes를 채택(유지)한다.**

## 근거

- 위 요구를 **검증된 방식으로 모두 충족**한다(역할 하이브리드 `model` + `delegation.model`, 백그라운드 실행, 메모리·스킬, `/v1`, deterministic 툴 — 본 대화에서 문서로 확인).
- **OpenClaw는 기능상 거의 동급**(서브에이전트 역할분할도 지원)이나, **결정적으로 이기는 요구가 없다.** 유일한 강점인 빌트인 음성(talk mode)도 우리가 어차피 **커스텀 오케스트레이터 + 커스텀 HUD 스트리밍**으로 직접 만들 영역이라 결정타가 아니다.
- 이미 **셋업·SOUL 페르소나·M1 앱(`/v1`) 연결 완료** → 전환 시 재셋업 비용·학습 리스크에 비해 능력 이득 0. 마감 앞에서 비합리.
- **자체 구축**(LangGraph/Letta/직접)은 메모리·스킬을 자가 구현하는 것 = 프로젝트가 명시적으로 피한 백엔드 늪 → North Star 위배.

## 대안 및 재검토 조건

- **OpenClaw** — near-peer. 우선순위가 "generative HUD"에서 "빠른 음성비서"로 이동하면 talk mode 때문에 재검토할 가치.
- **자체 구축 / 공유메모리 듀얼(Mem0)** — 코히어런스는 최상이나 늪. **데모 후** 더 깊이 갈 때의 옵션.

## 결과

- 모델 구성: 메인 = fast 비추론(Claude Haiku 4.5 / GPT-5.4 mini), delegation = strong(Claude Opus 4.6 / GPT-5.5).
- TTFT 실측은 생략(전부 클라우드 → 측정 이득 낮음).
- 다음 단계: **Hermes 실현가능성·세팅 감사**(ADR 0002) → 미확인 항목 de-risk 스파이크.
